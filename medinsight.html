System Architecture

Four-layer architecture designed for scalability, extensibility, and production reliability.

High-Level Architecture

üîå Plugin Module Architecture

Every extraction domain (quality measures, diagnostic coding, clinical highlights, etc.) is implemented as a self-contained plugin module. Each plugin defines its own LLM prompts, ML model configurations, validation rules, and output schemas ‚Äî registered through a Factory Pattern that allows new domains to be added with zero changes to the core engine.

üß©

Plugin Registration

Each domain plugin self-registers with the factory. The core engine dynamically loads extraction modules based on document type and configuration.

üîÑ

Shared Infrastructure

OCR, document parsing, queue management, and API endpoints are shared. Plugins only define domain-specific extraction logic.

üì¶

Isolated Deployment

Plugins can be enabled/disabled independently. Deploy new extraction domains without redeploying the entire platform.

üîß Adding a New Domain: What It Takes
1

Define Extraction Schema

Specify what data points to extract, their types, and validation rules via JSON configuration.

2

Create LLM Prompts

Design domain-specific prompts that instruct the LLM how to identify and extract target information.

3

Train ML Models (Optional)

For domains requiring classification (like diagnostic coding), train supervised models on labeled data.

4

Build Question Library

Create predefined Q&A templates for the "Ask the Document" feature. Free-form queries work immediately.

5

Register Plugin

Register the new domain with the factory. The core engine handles all processing, storage, and UI automatically.

6

Deploy & Validate

Zero-downtime deployment. Run validation against sample documents and calibrate extraction accuracy.
